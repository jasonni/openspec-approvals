## 背景脈絡

本專案已提供 OpenSpec approvals dashboard，包含專案選擇、變更列表、artifact 詳情頁、核准送出、語意搜尋，以及以 SSE 推送的即時更新。不過目前 UX 仍以審查流程為主，尚未提供完整的儀表板導向 spec 操作體驗（例如總覽指標、更完整的任務管理、明確的規格導覽流程）。

## 目標 / 非目標

**目標：**

- 提供包含專案層級指標與近期活動的儀表板總覽。
- 提供以 requirements、design、tasks 文件為中心的規格詳情導覽體驗。
- 提供由 `tasks.md` 衍生的階層式任務管理與進度視覺化。
- 提供更清楚的即時連線/更新狀態可視性。
- 定義儀表板導覽的鍵盤可及與響應式行為。

**非目標：**

- 不替換既有的核准資料儲存或 GitHub sync 資料模型。
- 不引入新的身分驗證/授權機制變更。
- 不重寫整體 UI 技術堆疊，也不替換 Next.js 路由架構。

## 設計決策

### 1) 在現有 OpenSpec reader 之上加入儀表板 view-model 層

實作上將在既有 `lib/` 服務與/或 API 路由中加入伺服器端彙整層，從目前 change artifacts 與 approval 資料計算出儀表板總覽數值。

- **選擇原因：** 可將較重的解析/彙整計算留在伺服器端，並重用既有資料存取函式。
- **評估替代方案：** 在前端從原始 API payload 全量計算。此方案因邏輯重複、前端渲染成本較高而不採用。

### 2) 新增標準 spec 導向路由，同時保留既有 change 路由

儀表板將支援 spec 導向路徑（例如 `/spec/<changeId>`、`/spec/<changeId>/requirements`、`/spec/<changeId>/design`、`/spec/<changeId>/tasks`），並保留既有 `/changes/<id>` 行為以維持向下相容。

- **選擇原因：** 深連結可提升可操作性，且 URL 結構更符合參考儀表板的使用預期。
- **評估替代方案：** 僅保留 `/changes/<id>` 的 query-param tab 切換。此方案可發現性較低、連結分享可讀性較差，因此不採用。

### 3) 將 `tasks.md` 解析為階層式任務模型

任務渲染將使用可預測的解析器處理編號任務（例如 `1.0`、`1.1`、`1.1.1`），並導出完成率與分段進度。

- **選擇原因：** 若無結構化任務模型，無法可靠支援進度條、依賴顯示與任務操作。
- **評估替代方案：** 僅渲染原始 markdown。此方案無法穩定提供進度視覺化與任務操作能力，因此不採用。

### 4) 將即時 UX 從被動事件列擴充為明確狀態 + 通知

即時傳輸仍沿用既有 SSE stream，但 UI 合約需擴充為可顯示連線狀態，並提供索引、核准、任務/規格異動的使用者通知。

- **選擇原因：** 使用者需要隨時判斷即時資料目前是否可信且可用。
- **評估替代方案：** 僅保留「最後事件」文字列。此方案操作可視性不足，故不採用。

### 5) 補齊儀表板層級的可及性與客製化基準

設計需涵蓋鍵盤導覽、焦點可視、語意標記，以及可跨工作階段保存的最小偏好模型（主題/顯示選項）。

- **選擇原因：** 儀表板屬高頻使用介面，必須可在不依賴指標裝置的情境下操作。
- **評估替代方案：** 將可及性與偏好設定延後。此方案會直接影響導覽效率與可讀性，不符合此變更核心目標，因此不採用。

## 風險 / 取捨

- **[Risk]** 各專案的 task markdown 格式可能不一致，未必符合嚴格編號假設。  
  **→ Mitigation:** 定義解析容錯規則，並對無法匹配的行提供回退顯示。

- **[Risk]** 額外的彙整/解析可能在大型儲存庫增加回應延遲。  
  **→ Mitigation:** 重用既有索引生命週期、在安全範圍導入快取，並限制單次請求的高成本計算。

- **[Risk]** 新增路由可能造成與既有連結的導覽不一致。  
  **→ Mitigation:** 保留向下相容路由，並提供路由正規化/重新導向規則。

- **[Risk]** 更豐富的即時通知可能增加介面噪音。  
  **→ Mitigation:** 定義通知嚴重度與去重行為，並納入 UI 狀態管理。

## 遷移計畫

1. 新增儀表板指標與結構化任務資料的型別合約與彙整輔助工具。
2. 引入新的 dashboard/spec 路由，並接入既有 `projectId` 範圍規則。
3. 逐步佈署總覽卡片、任務面板與連線狀態等 UI 元件。
4. 驗證既有審查與核准流程的相容性。
5. 為舊連結與不完整任務格式提供回退行為。
